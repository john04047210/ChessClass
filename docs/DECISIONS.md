# 关键设计决策

本文件记录不应在每次讨论中重新推翻的项目决策。若以后改变决策，请追加日期、原因和迁移影响，不要只删除旧结论。

## D-001：静态优先、免费优先

- 状态：采用
- 决策：默认产品必须能够作为纯静态网站运行，不依赖应用服务器。
- 原因：降低部署和维护成本，支持长期免费游玩，并让 GitHub Pages、对象存储和普通静态服务器都能承载。
- 影响：不能依赖 Next.js Server Actions、API Routes、服务器会话或服务端数据库完成核心功能。

## D-002：远程 AI 是可选增强

- 状态：采用
- 决策：没有 OpenAI 或其他 AI 配置时，下棋、电脑对手、进度和基础教学必须可用。
- 原因：避免统一 API Key 带来的费用失控，也避免网络成为核心玩法的前置条件。
- 影响：浏览器客户端只允许配置网关 URL，不能保存部署者的服务端密钥。玩家自行提供 API Key 的方案在解决浏览器密钥暴露问题前不作为默认架构。

## D-003：chess.js 是规则事实来源

- 状态：采用
- 决策：所有走法合法性和棋局结束状态由 `chess.js` 验证。
- 原因：避免界面、电脑引擎和教练各自实现棋规造成不一致。
- 影响：Stockfish 或本地启蒙引擎返回的着法仍需通过 chess.js 验证后才能落子。

## D-004：Stockfish Lite 为默认引擎，启蒙引擎永久保留

- 状态：采用
- 决策：默认使用 Stockfish 18 Lite 单线程 WASM，同时保留 `BeginnerOpponentEngine`。
- 原因：Lite 版本约 7 MB，棋力足够，兼容无需特殊响应头的静态托管；备用引擎适合更弱对手、旧设备和加载故障。
- 影响：Stockfish 在 Web Worker 内运行；界面允许玩家选择引擎；加载失败必须自动降级。

## D-005：用户数据本地优先

- 状态：采用
- 决策：匿名档案、棋局、时间线、教练内容和设置保存在 localStorage。
- 原因：无需账号和服务器，符合免费和隐私优先方向。
- 影响：数据只存在当前浏览器；结构变更必须向后兼容；更换设备和清除浏览器数据会丢失进度。

## D-006：悔棋是可浏览的线性时间线

- 状态：采用
- 决策：玩家可以在未重新走棋前沿历史向后或向前浏览；在历史节点重新走棋时截断其后的旧分支。
- 原因：悔棋功能的目标是帮助玩家反复复盘，而不是只有一次撤销。
- 影响：最多保留初始节点加 100 个完整回合节点，每个节点保存棋局和对应教练快照。

## D-007：教学陈述必须区分事实和推测

- 状态：采用，尚未完整实现
- 决策：将军、吃子、攻击、保护、开线等可计算内容按事实表达；战略意图使用“可能”“看起来”等措辞并说明依据。
- 原因：无论模板还是大语言模型都可能误判走棋动机，初学者不应把猜测当成棋规事实。
- 影响：下一代本地教练需要输出结构化证据和置信度，不能继续使用万能原因模板。

## D-008：三语功能保持一致

- 状态：采用
- 决策：简体中文、英文和日文共享相同功能与领域数据，语言文件只负责展示文案。
- 原因：避免某种语言获得不同规则或行为。
- 影响：文案 key 必须三语同步；FEN、PGN、SAN、RuleKey 等领域数据保持语言无关。

## D-009：GitHub Pages 自动部署

- 状态：采用
- 决策：`main` 由 GitHub Actions 使用 Node.js 24 构建，并发布 `out/` 到 `/ChessClass`。
- 原因：部署免费、可重复，不需要在源码分支提交生成文件。
- 影响：修改资源路径或路由时必须验证 Pages base path；`out/` 不进入 Git。

## D-010：对话与棋局模型为未来人人对战保持颜色和来源中立

- 状态：采用，预留但尚未实现人人模式
- 决策：`player`、`opponent` 是相对当前观看者的展示角色，不与白方、黑方或 Stockfish 永久绑定；走棋数据显式保存 participant、color 和 source。
- 原因：未来计划增加本地双人和在线人人对战。如果本次对话重构把玩家写死为白方、对手写死为黑方引擎，后续需要再次重写消息、棋局和时间线。
- 影响：引擎分析消息只在引擎参与或赛后复盘时出现；在线实时教练默认关闭；在线棋局未来必须由服务端维护权威状态，不能把当前 localStorage 时间线直接当作网络协议。
